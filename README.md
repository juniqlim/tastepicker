# tastepicker

내 취향에 맞는 큐레이터를 내가 고르고, 그가 고른 것을 본다.

평점과 랭킹은 다루지 않는다. 다수의 판단이 아니라 내가 신뢰하는 소수의 판단만 모은다.

## 원칙

- **픽커는 내가 임명한다.** 서비스는 후보를 추천할 수 있어도 목록에 자동으로 넣지 않는다.
- **출처를 밝힌다.** 모든 픽은 픽커와 원문 링크를 반드시 가진다.
- **원문은 픽커의 것으로 보여준다.** 서비스가 쓴 글처럼 내보이지 않는다. 인용은 발췌로 하고 원문으로 넘어갈 길을 남긴다.
- **많이보다 확실하게.** 근거가 불분명한 데이터는 버린다.

## 용어

| 용어 | 뜻 |
| --- | --- |
| picker | 내가 검증해서 등록한 큐레이터 |
| pick | 픽커의 한 번의 방문 |
| place | 가게. 네이버 장소 ID로 잡는다 |

한 줄은 한 번의 방문이다. 같은 가게를 다시 가면 줄이 하나 더 생긴다.
갈 때마다 점수와 한줄평이 달라질 수 있고, 그 자체가 정보다.
정직한 청년은 할아버지 참나무 장작구이를 네 번 갔다. 등급을 안 매겨도 재방문이 곧 평가다.

가게는 이름이 아니라 네이버 장소 ID로 묶는다.
픽커마다 `장작구이`, `할아버지참나무장작구이` 로 달리 적어도 같은 집이다.
화면에 보이는 이름은 네이버 상호로 통일한다. 안 그러면 남이 볼 때 다른 집으로 보인다.

픽의 공통 항목은 `picker`, `link`, `level` 셋이다. 나머지는 픽커가 정한다.
맛집 픽커는 `region`·`name`을, 투자 픽커라면 다른 항목을 담는다.

## 평가

픽커마다 평가하는 말이 다르다. 그 말은 `rating`에 원문 그대로 둔다.
픽커끼리 견주려면 공통 자가 필요해서 `level` 하나로 옮긴다.

| level | 이름 | RockHer의 말 |
| ---: | --- | --- |
| 5 | 강추 | 강추 |
| 4 | 추천 | 추천 |
| 3 | 보통 | 괜춘·쏘쏘·보통·평범·무난 |
| 2 | 그닥 | 그닥 |
| 1 | 비추 | 별로 |

눈금을 5로 둔 건 데이터가 그만큼만 구분하기 때문이다. 10단계로 두면 절반이 빈다.

어떻게 옮겼는지는 `levelBy`에 남긴다. 픽커가 매긴 등급을 옮긴 것과 우리가 한줄평에서 추측한 것은 무게가 다르다.
정직한 청년은 등급을 매기지 않아 지금은 비어 있다.

## 현재

네이버 블로그 두 곳의 글 전체에서 맛집 픽을 모은다.

```
src/request.js   천천히 부르고 막히면 물러섰다 다시 부른다
src/naver.js     글 목록과 본문 받아오기
src/rss.js       RSS → 글 목록
src/pickers.js   글 → 픽 (픽커마다 규칙 하나)
src/place.js     본문 → 픽커가 붙인 장소
src/db.js        SQLite 저장
```

픽커마다 제목 형식이 달라 범용 파서를 두지 않는다. 소수만 다루므로 규칙을 하나씩 쓰는 쪽이 정확하다.
제목 규칙만으로 충분히 걸러져서 카테고리는 보지 않는다.

| 픽커 | 형식 | 전체 글 | 픽 |
| --- | --- | ---: | ---: |
| [정직한 청년](https://blog.naver.com/thddbcjf) | `지역 가게명-한줄평` | 1,560 | 766 |
| [RockHer](https://blog.naver.com/fascinoya) | `[지역 분류] 가게명 (등급) - 설명` | 4,381 | 1,552 |

픽커가 꼭 블로그일 필요는 없다. `juniqlim`은 내 평가라 긁어올 글이 없다.

## 내 평가

블로그 픽은 안 변하니 빌드할 때 HTML에 박고, 내 평가는 자주 바뀌니 Supabase에 두고 열 때 받아온다.

```
블로그 픽 → data/picks.db → 빌드 때 HTML에 박음
내 평가   → Supabase      → 페이지 열 때 조회
```

안 변하는 것을 매번 조회할 이유가 없다. Supabase가 멈춰도 지도는 뜬다.

읽기는 누구나, 쓰기는 나만 할 수 있게 막아 두었다(`data/supabase.sql`).
`anon` 키는 브라우저에 들어가므로 공개되지만, 로그인하지 않으면 쓰기가 거부된다.
아직 로그인을 붙이지 않아 평가 추가는 서버 쪽에서만 한다.

가게는 픽커들이 이미 다녀간 곳에서 고른다. 그래야 겹치는 가게가 생겨 견줄 수 있다.

RockHer는 본인이 등급을 붙인 글만 받는다. 등급 없는 글을 추천으로 볼 근거가 없다.
등급은 좋은 순으로 `강추` `추천` `괜춘` `쏘쏘` `보통` `평범` `무난` `그닥` `별로` 아홉 가지다.
나쁜 평가도 픽커의 판단이라 함께 받는다. 오히려 이 픽커가 광고 글을 쓰지 않는다는 증거다.

한 글에 여러 가게를 쓰기도 해서 첫 등급 앞까지를 가게명으로 본다.

## 저장

`data/picks.db` (SQLite). Node에 내장돼 있어 따로 붙일 것이 없다.
글 주소가 열쇠라 다시 받아도 쌓이지 않고 갱신된다. 받다 끊기면 이어서 받는다.
장소는 한 번만 받아둔다. 제목 규칙을 고쳐 다시 돌려도 본문을 또 받지 않는다.
규칙에서 빠진 글은 지워서 DB가 규칙과 어긋나지 않게 한다.

읽기만 하는 데이터라 이걸로 충분하다.
웹에서 픽커를 등록하거나 사용자가 여럿이 되면 그때 옮긴다.

## 매일 수집

매일 새벽 5시에 `npm run build` 를 돌려 새 글을 모은다(`.github/workflows/collect.yml`).

```
새 글 수집 → data/picks.db 커밋 → push → Vercel 재배포
```

목록은 전체를 훑고 본문은 새 글만 받아서 1분 반이면 끝난다.
따로 이어받기 표시를 두지 않는 건, 픽커의 옛 글이 지워지거나 제목이 바뀌어도 DB가 따라가게 하기 위해서다.

새 글이 없으면 커밋하지 않아 배포도 일어나지 않는다.
새 글이 있는지는 `data/picks.sha`(픽 전체의 지문)로 본다.
SQLite 파일은 내용이 같아도 바이트가 달라져서, 파일만 보면 매일 840KB가 히스토리에 쌓인다.

## 좌표

픽커가 글에 붙인 네이버 장소에서 그대로 가져온다.
가게명으로 검색해 추측하지 않는다. 픽커가 지목한 곳이 정답이고 동명 가게 문제도 없다.

```json
{ "placeId": "1209575927", "name": "억떡볶이",
  "address": "경기도 안양시 동안구 평촌대로 127 중앙상가 1층",
  "lat": 37.3833212, "lng": 126.9595726, "tel": "0507-1306-4057" }
```

## 지도

`public/index.html` 에 찍는다. **핀 하나는 가게 하나다.** 픽 하나가 아니다.
한 가게를 여러 픽커가, 또 같은 픽커가 여러 번 쓰기 때문에 픽마다 찍으면 겹쳐서 하나만 눌린다.
핀을 누르면 그 가게에 달린 평가가 방문 순으로 모두 나오고, 줄마다 원문 링크가 붙는다.

등급은 픽커마다 다르다. RockHer는 아홉 단계를 쓰고 정직한 청년은 매기지 않는다.
한 픽커의 등급 체계를 전체 축으로 삼을 수 없다.

그래서 색은 모든 픽이 가진 픽커로 나누고, 등급은 같은 색의 진하기로만 보인다.
켜고 끄는 것도 픽커 아래에 그 픽커의 등급을 둔다.

해외 픽이 섞여 있어 전체로 맞추면 세계 지도가 열린다. 국내 픽 기준으로 연다.
지금은 OpenStreetMap으로 미리 본다. 좌표만 있으면 되므로 네이버 지도로 바꿔도 데이터는 그대로다.
